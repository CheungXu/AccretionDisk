# Sprout 前端优化与工作流编排

## 背景

在 `sprout` 萌芽项目一期后端能力（项目导入、节点执行、版本管理、API）完成的基础上，为了更好地服务于创意从业者，我们需要对前端工作台进行视觉与交互的优化，并提供更直观的工作流管理能力。

## 本次优化目标

1. **视觉升级**：引入充满活力的色彩搭配、毛玻璃效果（Glassmorphism）和流畅的过渡动画。
2. **工作流可视化**：在首页展示整个项目的工作流管线（Workflow Diagram），支持状态指示（成功、运行中、失败）。
3. **定制化节点详情**：针对不同类型的节点（如剧本、角色、镜头、导出等），提供直观、美观的卡片式内容载荷展示，替代原始的 JSON 视图。
4. **前端工作流编排**：支持“一键启动工作流”，由前端依次调用未完成节点的 API，实现自动化执行。

## 本轮补充目标

在完成首版工作台之后，本轮进一步补充了更贴近真实创作流程的能力：

1. **工作流新增起始用户输入节点**：把题材、时长、镜头数、画幅、风格、受众与已有分镜文本纳入工作流与版本管理。
2. **导入交互升级为目录选择器**：不再让用户手填路径，而是通过本机原生目录选择器导入项目。
3. **空目录可初始化为空项目**：当用户选择空目录时，系统自动创建最小占位项目，再通过用户输入节点继续规划。
4. **新增脚本分镜展示节点**：与角色资产并行展示剧情、脚本、分镜与分镜图片，便于创作者在角色图生成前后同步审查内容结构。
5. **修复镜头节点显示为空的问题**：节点详情页和工作台中的镜头描述、提示词等内容改为读取真实字段。

## 架构决策：前端编排工作流

在实现“一键启动”功能时，我们面临两种选择：
1. 后端新增 `run-all` 接口，在后端维护执行状态。
2. 前端根据项目节点列表，依次调用现有的单节点执行 API。

**最终决策**：采用**前端编排**（Frontend Orchestration）。
- **原因**：当前后端已经提供了完善的单节点执行能力和状态查询能力。前端编排可以最大程度复用现有 API，避免后端引入复杂的异步任务管理和状态推送机制（如 WebSocket）。同时，前端编排能更方便地在界面上实时更新每个节点的执行进度和管线状态。

## 核心实现细节

- **样式重构**：`app.css` 中定义了全新的 CSS 变量体系，使用了深色模式下的渐变背景和半透明面板，提升了整体质感。
- **管线渲染**：在 `renderers.js` 中新增 `renderWorkflowDiagram`，根据 `projectDetail.nodes` 动态生成带有连接线的节点图示。
- **内容载荷定制**：新增 `renderCreativePayload` 方法，将原本枯燥的 JSON 数据转换为结构化的卡片视图（如角色设定卡片、镜头提示词高亮等）。
- **一键启动逻辑**：在 `index.js` 中实现 `handleRunWorkflow`，遍历节点列表（跳过通常需要人工干预的 `plan` 节点），依次调用 `runNode`，并在每个节点完成后重新拉取项目详情以更新 UI。

## 本轮实现决策

### 1. 用户输入成为版本源节点

- `user_input` 被设计为工作流首节点。
- 每次执行 `user_input` 都会生成独立版本快照，后续节点的版本依赖从这一层开始向下传递。
- 这样在切换题材、镜头数或分镜文本后，`角色资产`、`提示词准备`、`视频生成` 等节点都能自动识别上游版本变化。

### 2. 脚本分镜作为并行展示节点

- `script_storyboard` 放在 `user_input` 后，与 `characters` 并行展示。
- 该节点不单独执行，直接复用 `user_input` 的激活版本。
- 这样既能让创作者看到“输入生成了什么剧情与镜头”，又不会给后端增加一套重复执行路径。

### 3. 空目录导入策略

- 如果目录已有 `sprout` bundle，则保持不变，按原项目忠实导入。
- 如果目录为空，则自动初始化一个最小占位项目，只保留 `user_input` 节点。
- 初始化完成后，用户可在页面中填写题材或已有分镜，再从该节点继续生成完整项目。

### 4. 镜头展示字段对齐

本轮确认并统一了前端镜头展示的字段来源：

- 画面描述：`visual_description`
- 首帧提示词：`keyframe_prompt`
- 视频提示词：`video_prompt`
- 分镜图片：优先展示 `shot_keyframe`

此前前端错误读取了 `description` / `prompt` 之类的旧字段，因此出现“节点里看起来是空的，但后端其实已有内容”的问题。

## 本轮交付结果

- 工作流已从单线性结构升级为“主链路 + 并行展示节点”结构。
- 首页导入方式已改为目录选择器，降低了普通用户的使用门槛。
- 空目录可直接作为新项目起点，避免必须先手工构造 bundle。
- 镜头节点详情中的剧情、描述、提示词和分镜图片已能正确展示。
- 一键启动逻辑已经改为基于最新项目状态迭代执行，空项目在生成出新节点后也能继续跑后续链路。

## 后续建议

- 随着节点类型的增加，可以进一步丰富 `renderCreativePayload` 的展示模板。
- 若未来工作流变得非常庞大且耗时极长，可能需要考虑将编排逻辑下沉到后端，并引入真正的异步任务队列。但对于当前的短剧生产链路，前端编排已足够高效且易于维护。
- 如果未来还会增加更多“只展示不执行”的内容节点，建议继续沿用“版本源节点 + 展示派生节点”的模式，避免节点树和执行树混在一起。
