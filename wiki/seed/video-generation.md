# 任务式视频生成接口的封装要点

## 适用场景

当视频生成接口不是同步返回结果，而是先创建任务、再轮询查询结果时，调用层需要额外处理任务生命周期。

典型流程：

1. 创建任务
2. 提取任务 ID
3. 轮询任务状态
4. 在成功状态下提取视频 URL
5. 按需下载到本地

## 推荐封装方式

建议将视频接口按任务式客户端封装，而不是只提供单次请求方法。

至少应包含：

- 创建任务
- 查询任务
- 等待完成
- 提取结果 URL
- 下载到本地

这样做的好处：

- 业务层调用更简单
- 重试、超时、状态判断可以集中处理
- 后续扩展更多任务场景时更一致

## 图生视频的常见坑

图生视频往往不只是“带一张图”，还涉及输入图片的角色定义。

实践中需要注意：

- 不同角色可能会影响后端将任务识别为哪一种任务类型
- 某些模型下，`reference_image` 可能会触发与预期不一致的任务分类
- 对明确的图生视频场景，通常更适合使用 `first_frame`

## 实践经验

在本次接入中，图生视频的稳定做法是：

- 图片角色默认使用 `first_frame`
- 明确指定 `task_type="i2v"`

这样可以降低后端将任务误判为其他模式的风险。

## 工程建议

如果视频生成结果需要进入后续流程，建议默认补齐以下能力：

- 任务状态轮询
- 超时控制
- 成功/失败状态显式判断
- 最终 URL 提取
- 本地下载
- 默认参数配置外置

## 可复用原则

这套思路不仅适用于视频生成，也适用于所有“先创建任务、后异步查询结果”的生成式接口：

- 视频生成
- 长耗时图片任务
- 音频生成
- 批处理生成任务

## 本次接入总结

本次 Seed 视频接入中，最终形成的稳定工程做法包括：

- 模型默认参数从 `config/seed_config.json` 读取
- API Key 从环境变量或 `config/seed_key.json` 读取
- 图生视频默认走 `first_frame + i2v`
- 任务完成后支持直接下载到本地
