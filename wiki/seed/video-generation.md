# 任务式视频生成接口的封装要点

## 适用场景

当视频生成接口不是同步返回结果，而是先创建任务、再轮询查询结果时，调用层需要额外处理任务生命周期。

典型流程：

1. 创建任务
2. 提取任务 ID
3. 轮询任务状态
4. 在成功状态下提取视频 URL
5. 按需下载到本地

## 推荐封装方式

建议将视频接口按任务式客户端封装，而不是只提供单次请求方法。

至少应包含：

- 创建任务
- 查询任务
- 等待完成
- 提取结果 URL
- 下载到本地

这样做的好处：

- 业务层调用更简单
- 重试、超时、状态判断可以集中处理
- 后续扩展更多任务场景时更一致

## 图生视频的常见坑

图生视频往往不只是“带一张图”，还涉及输入图片的角色定义。

实践中需要注意：

- 不同角色可能会影响后端将任务识别为哪一种任务类型
- 某些模型下，`reference_image` 可能会触发与预期不一致的任务分类
- 对明确的图生视频场景，通常更适合使用 `first_frame`

结合官方文档，当前可进一步明确：

- 单主图图生视频时，传入 `1` 个 `image_url` 即可，`role` 为 `first_frame` 或不填
- 若传入多张额外参考图，则每张参考图的 `role` 应为 `reference_image`
- 官方推荐在提示词中使用 `"[图1]...，[图2]..."` 这种写法，提高多图指令绑定效果
- 官方文档中还定义了首尾帧场景：传 `2` 个 `image_url` 时，分别使用 `first_frame` 与 `last_frame`

## 实践经验

在本次接入中，图生视频的稳定做法是：

- 图片角色默认使用 `first_frame`
- 明确指定 `task_type="i2v"`

这样可以降低后端将任务误判为其他模式的风险。

## 工程建议

如果视频生成结果需要进入后续流程，建议默认补齐以下能力：

- 任务状态轮询
- 超时控制
- 成功/失败状态显式判断
- 最终 URL 提取
- 本地下载
- 默认参数配置外置

如果涉及有声/无声或多参考图，建议额外关注：

- `generate_audio` 是官方标准参数，默认值为 `true`
- `generate_audio` 的官方支持范围包含 `Seedance 2.0`、`Seedance 2.0 fast`、`Seedance 1.5 Pro`
- 多参考图并不是所有模型都统一支持，官方明确列出的主要是 `Seedance 2.0`、`Seedance 2.0 fast`（`1~9` 张）以及 `Seedance 1.0 lite i2v`（`1~4` 张）
- 若项目默认模型不在官方多参考图支持列表内，应将“接口可传”与“模型能力已确认”区分开

## 可复用原则

这套思路不仅适用于视频生成，也适用于所有“先创建任务、后异步查询结果”的生成式接口：

- 视频生成
- 长耗时图片任务
- 音频生成
- 批处理生成任务

## 本次接入总结

本次 Seed 视频接入中，最终形成的稳定工程做法包括：

- 模型默认参数从 `config/seed_config.json` 读取
- API Key 从环境变量或 `config/seed_key.json` 读取
- 图生视频默认走 `first_frame + i2v`
- `generate_audio` 可通过配置或调用参数控制有声/无声输出
- 多参考图入参默认按 `reference_image` 组织，但正式上线前应先核对目标模型是否在官方支持范围内
- 任务完成后支持直接下载到本地
