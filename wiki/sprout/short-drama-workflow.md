# AI 短剧萌芽工作流

## 适用目标

适用于从一句题材或已有分镜出发，快速生成 AI 短剧的角色资产、镜头视频与交接清单。

## 基础流程

1. 明确题材与时长
2. 生成标题、爽点、角色设定、镜头表
3. 先固定主角色形象
4. 再按镜头逐条生成视频
5. 最后做剪辑拼接与复核

## 官方能力对齐

### 有声视频

- `generate_audio` 是官方标准参数
- 适合把台词和音频要求直接纳入镜头生成链路

### 多图参考绑定

- 官方推荐在 prompt 中使用 `"[图1] ... [图2] ..."` 的写法
- 工程上必须维护“上传顺序”和“编号顺序”一致
- 主图通常作为 `first_frame`
- 额外参考图通常作为 `reference_image`

## 工程上的关键点

### 角色先行

- 角色图不是装饰图，而是后续所有镜头的一致性锚点
- 如果人物复杂，建议补充多角度参考图

### 镜头拆分

- 短剧生产更适合按镜头逐段生成
- 每个镜头应保留画面描述、角色表、台词、时长和绑定信息

### prompt 结构

推荐把 prompt 拆成三层：

1. 参考图绑定层：`[图1]`、`[图2]` 对应谁
2. 画面描述层：人物动作、机位、情绪、场景
3. 音频补充层：台词、语气、环境音要求

## 一期建议

- 先把 `一句题材 -> 一个镜头跑通` 做实
- 再扩展到 `10 镜头 -> 1 集`
- 剪辑自动化可以放到后续阶段，避免一开始把链路做得太重

## 节点式调用建议

如果需要边调边测，推荐按下面顺序分步调用：

1. `plan-topic` 或 `plan-storyboard`
2. `build-characters`
3. `prepare-shots`
4. `generate-shots`
5. `build-cards`
6. `export`

这样做的好处：

- 每一步都会把项目状态保存到 bundle JSON
- 出问题时可以从某一步继续，而不是整条链路重跑
- 更适合单镜头、小范围提示词调试

## 当前工程实现

当前 `sprout` 已经将这套流程拆成了独立模块：

- `script_planner`：输出结构化项目包
- `character_builder`：输出角色锚点图与补图
- `shot_pipeline`：输出关键帧 prompt、`[图N]` 绑定、有声视频
- `jimeng_packager`：输出即梦执行卡
- `exporter`：输出 bundle、manifest、summary 和执行卡文件

## 账号能力差异的兜底

实际接入时，多参考图视频不只取决于代码是否支持，还取决于账号是否拿到对应模型权限。

当前工程兜底策略：

- 优先尝试 `Seedance 2.0` 系列多参考图模型
- 若模型不存在或账号无权限，则回退为：
  1. 先用多参考图生关键帧
  2. 再用单图 `i2v` 生成有声视频

这样做的价值：

- 单节点测试仍然能继续跑
- 不会因为模型权限问题把整条链路卡死
- 后续一旦账号开通更高能力，只需要改 CLI 参数，不必重写流程

## 工作台实现建议

当 `sprout` 从 CLI 演进到工作台时，建议明确拆成三层：

- `core/`：只负责剧本规划、角色生成、镜头准备、镜头生成、执行卡与导出
- `service/`：负责项目导入、项目注册表、运行时日志、版本快照、媒体访问和 API
- `web/`：负责页面交互，只通过 API 读取项目、节点、版本与日志

当前工程已经按这条路线收敛：

- 核心工作流真实实现只保留在 `core/`
- `service/` 负责工作台后端能力
- `web/` 负责工作台页面
- 顶层业务模块已删除，不再保留第二套实现路径

这样做的价值：

- 核心工作流仍能继续被 CLI 和脚本复用
- 后端可以承接“节点级版本切换”和“热插拔继续跑”
- 前端不直接耦合磁盘目录，后续扩到其他 AIGC 项目时更容易复用
- **前端编排工作流**：前端可以根据返回的节点列表，实现可视化管线图示并提供“一键启动”功能，依次调用各节点接口，从而降低后端状态管理的复杂度。

## 工作台节点设计建议

### 版本源节点与展示节点要分开

如果工作台里既有“真正会执行并生成新结果的节点”，也有“只负责把已有内容展示给创作者看的节点”，建议将两类节点显式拆开：

1. **版本源节点**：例如 `user_input`，负责接收用户输入并生成新的版本快照
2. **执行节点**：例如 `characters`、`prepare_shot`、`generate_shot`
3. **展示节点**：例如 `script_storyboard`，只复用某个上游版本，不单独执行

这样做的好处：

- 版本链清晰，不会出现两个节点重复生成相同快照
- UI 可以展示并行信息结构，但后端执行路径仍保持简单
- 当上游输入切换版本时，展示节点可以自动同步，不需要额外维护第二套状态

### 节点依赖与视觉布局要分开

工作流在工程上至少有两种“关系”：

1. **执行依赖关系**：谁完成后谁才能继续执行
2. **视觉布局关系**：谁应该并排展示、谁应该在主链路上展示

建议在节点定义里分别保留：

- `upstream_node_ids`：真实执行依赖
- `visual_parent_node_id`：界面上的挂载关系
- `version_source_node_id`：版本实际来源

这样可以支持如下场景：

- `user_input -> characters -> prepare_shot`
- `user_input -> script_storyboard`

其中 `script_storyboard` 可以跟 `characters` 并列显示，但并不打断主执行链。

## 节点版本管理建议

如果工作台需要支持“重跑但保留旧结果”和“从某个中间版本继续跑后续节点”，建议：

1. 每次节点执行都生成一个版本快照
2. 当前项目保留一个“激活版本”视图，兼容原有 bundle 读取
3. 运行日志单独落盘，不和 bundle 混写
4. 图片、视频等产物仍按项目目录归档，但运行记录和版本索引进入 `runtime/`

## 前端字段对齐建议

工作台页面不要假设后端一定会保留“简化字段名”。

短剧项目里，镜头节点至少应优先读取这些真实字段：

- 画面描述：`visual_description`
- 首帧提示词：`keyframe_prompt`
- 视频提示词：`video_prompt`
- 台词：`dialogue`
- 音效：`sound_effects`
- 机位与镜头语言：`camera_language`

如果前端错误读取成 `description`、`prompt` 之类的历史字段，会出现“后端已有内容，但页面显示为空”的假象，影响创作者判断链路是否正常。

## 空项目导入建议

当工作台面向真实用户时，导入一个完全空的目录是高频需求。建议：

1. 允许空目录导入
2. 自动初始化最小占位项目
3. 只保留 `user_input` 起始节点
4. 待用户填写后再生成完整 bundle 与后续节点

这样比要求用户先手工准备 bundle 更自然，也更符合“从灵感开始”的工作流心智。
