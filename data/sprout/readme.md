# sprout 数据目录说明

`data/sprout/` 用于保存 `sprout` 项目的输入数据、中间产物和最终输出。

**说明：** 项目运行时数据（注册、版本、运行记录、快照等）已统一存放在 Supabase 云端数据库与 Storage；本目录仅在本地用于 CLI 单节点测试等场景，不再作为生产态权威数据源。

## 建议结构

- `input/`：原始题材、原始分镜、补充说明
- `script/`：结构化剧本、镜头表、人物卡
- `characters/`：角色定妆图、补充参考图、角色映射信息
- `shots/`：镜头级关键帧、镜头 prompt、镜头状态信息
- `videos/`：生成后的视频结果
- `workflow_cards/`：即梦执行卡、网页端备用提示
- `manifest/`：项目总清单、镜头索引、交接文件

## 管理原则

- 所有项目数据建议按项目名或日期继续分目录。
- 角色图与镜头视频需要保持稳定命名，便于 `[图N]` 绑定和结果复用。
- 若后续接入批量处理，优先让输出结构可断点续跑、可追溯。
- 若接入工作台，项目目录建议保留 `script/` 与 `manifest/`，便于导入识别。

## 工作台接入约定

- 当前 `sprout` 工作台会优先识别符合本目录约定的项目结构。
- 推荐将真实项目放在稳定目录下，避免导入后频繁移动根目录。
- 工作台现在支持原地引用项目目录，也支持复制到标准目录后再纳管。

## 当前真实测试目录

当前已使用 `data/sprout/node_test_case/` 做过一轮真实单节点调用测试，目录下已包含：

- `input/`：题材输入
- `script/`：结构化 bundle
- `characters/`：3 个角色的人设图与补图
- `shots/`：第 1 个镜头关键帧
- `videos/`：第 1 个镜头视频
- `workflow_cards/`：10 个镜头执行卡
- `manifest/`：项目 summary 与 manifest

## 单节点复用建议

- 每跑完一个节点后，优先复用 `script/{project_name}_bundle.json`
- 下一节点直接读取 bundle，避免重复跑前序链路
- 若账号缺少 `Seedance 2.0` 多参考图权限，当前工程会自动回退为“多图生关键帧 + 单图图生视频”
